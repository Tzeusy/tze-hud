# relay_resource_url Agent Exposure — Security Verdict

**Bead:** hud-04043
**Status:** Decision record — OMIT
**Date:** 2026-04-24
**Origin:** RFC 0018 §10 Open Question 5 — deferred at PR #548 review; resolved here
**RFC amended:** `about/legends-and-lore/rfcs/0018-whip-signaling-adapter.md` (Amendment A3)

---

## Question

RFC 0018 §10 OQ5 defers the following:

> **`relay_resource_url` confidentiality.** The WHIP resource URL is a capability
> token — sending it to the agent (`CloudRelayOpenResult.relay_resource_url`) for
> chrome display is a minor information exposure (the URL alone does not grant access
> without the bearer token, but it reveals the SFU endpoint structure). Consider
> omitting from the agent-facing message and retaining only in operator-facing
> telemetry. Deferred: include for now, revisit at security review.

**Verdict: OMIT.**

Remove `relay_resource_url` from the agent-facing `CloudRelayOpenResult` proto
message. Retain only in operator telemetry (audit log, already hashed per §7.3
`cloud_relay_drop` event schema).

---

## Tradeoff Analysis

### What does the agent actually need from `relay_resource_url`?

Nothing functional. The agent has all it needs from these fields already present in
`CloudRelayOpenResult`:

| Field | Agent use |
|---|---|
| `established` | Did the relay path come up? |
| `sdp_answer` | SDP answer to complete offer/answer exchange |
| `relay_epoch` | Stable relay path ID for reconnect correlation |
| `close_reason_code` / `close_reason_detail` | Failure diagnosis |

The RFC's proto comment on field 5 says "for operator-level audit and chrome display
only" and "Agents MUST NOT send HTTP PATCH or DELETE to this URL directly — the
runtime owns the relay path lifecycle." The agent has no protocol operation that
requires the URL. The "chrome display" use case — showing the relay endpoint URL
somewhere in UI chrome — is weak and unspecified. No concrete agent interaction
depends on it.

For relay reconnect, the agent uses `relay_epoch` (not the URL). After receiving
`CloudRelayCloseNotice`, the agent issues a fresh `CloudRelayOpen`; the runtime
picks the endpoint again from config. The URL is not carried forward across
reconnects.

### What does exposure leak?

Even granting that the WHIP resource URL does not grant access without the bearer
token, it leaks:

1. **SFU vendor identity.** LiveKit resource URLs follow `/rtc/whip/{room}/{uuid}`
   path shapes; Cloudflare Realtime session URLs follow `/sessions/{sessionId}`
   shapes. The pattern reveals which SFU vendor the operator selected — a commercial
   sensitivity in multi-tenant deployments.

2. **Endpoint topology.** Regional endpoint selection is reflected in the URL
   (subdomain, path). An agent that receives this across multiple relay sessions
   can infer regional routing topology.

3. **Session handle for CF Realtime.** On the Cloudflare Realtime adapter path, the
   "resource URL" is the session/tracks management handle — the active session
   identifier in CF's REST API. Its exposure is not merely cosmetic; it maps more
   directly to a session control surface than a WHIP Location URL does for LiveKit.

4. **Capability-token semantics.** §9.3 notes the resource URL is "UUIDv4 or
   equivalent random token generated by the SFU." It behaves like a capability
   token: possession plus a valid bearer token (already held by the runtime) is
   sufficient to mutate relay state. Agents have no need to hold this.

### Does the "chrome display" use case justify exposure?

No. The original RFC author flagged it as a hypothetical, not a specified
requirement. Agent UI chrome for cloud-relay status is fully served by:

- `relay_epoch` — relay path identity
- `established` — relay active/failed
- `CloudRelayStateUpdate` — RTT, packet loss, relay bitrate (already a separate
  server message, field 82)

None of these require the raw resource URL. If an operator-facing status panel ever
needs the endpoint domain for display, that is an operator telemetry concern, not
an agent-facing field.

### Does the CF proprietary path change the calculus?

It makes OMIT more compelling. The CF Realtime adapter's "resource URL" is a live
session management handle (POST to `/sessions/new` returns a `sessionId` used
in all subsequent renegotiate calls). Unlike a WHIP Location URL — which the
runtime uses only for PATCH/DELETE — the CF handle represents ongoing session state
with broader mutation surface. Passing this to the agent has no upside beyond the
weak "chrome display" rationale.

### Hardening alternatives considered

| Option | Verdict |
|---|---|
| **KEEP** | Rejected. No agent-functional need. Exposure risk not justified. |
| **REDACT** (return opaque handle) | Rejected. An opaque handle has zero agent utility — it cannot be displayed meaningfully, and the agent has `relay_epoch` for correlation. Adding a new opaque field achieves nothing and complicates the proto. |
| **OMIT** | Adopted. Clean removal. Operator telemetry (audit log) retains the full URL (hashed in `cloud_relay_drop` event per §7.3; unhashed in internal operator logs). No agent capability is lost. |

---

## Decision: OMIT

Remove `relay_resource_url` (field 5) from `CloudRelayOpenResult`.

The runtime retains the URL internally for:
- WHIP PATCH and DELETE operations on the relay path lifecycle
- Operator audit log (`cloud_relay_drop`, `cloud_relay_teardown`, `cloud_relay_activation` events — URL hashed in agent-correlated events, full URL in operator-only log sink per RFC 0019)

The field is marked `reserved 5;` in the proto to prevent field number reuse.

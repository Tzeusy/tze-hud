# V1 Scope

The manifesto describes the full vision. V1 is not the full vision. V1 is the smallest system that proves the thesis — that LLMs can have live, governed, performant presence on a screen — and that is worth building the rest on top of.

This document defines what v1 ships and what it explicitly defers.

## V1 must prove

1. **An LLM can hold a tile on a screen.** Create a tile, put content in it, update it over time, and have it render at 60fps with correct layout and z-order.

2. **The lease model works.** Agents authenticate, receive capability grants, hold leases with TTL, get revoked cleanly. The runtime is sovereign.

3. **Multiple agents coexist.** At least 3 concurrent resident agents sharing the screen, each with their own tiles and leases, without interfering with each other.

4. **Performance is real.** p99 frame time under 16.6ms. Split latency budgets: input_to_local_ack p99 < 4ms, input_to_scene_commit p99 < 50ms, input_to_next_present p99 < 33ms. Not as aspirations — as tested, measured, hardware-normalized properties of the shipping system.

5. **The validation architecture works.** All five layers operational. An LLM can build, test, iterate, and land changes autonomously. Developer visibility artifacts are generated on every PR.

6. **Zones work as the LLM-first surface.** An LLM can publish content to a zone via a single MCP tool call with no prior scene context, and the content renders correctly with the zone's geometry and rendering policy.

7. **Headless mode is fully functional.** The entire compositor can run without a display server. CI runs on software GPU.

## V1 ships

### Scene model
- Tabs (create, switch, delete)
- Tiles (create, resize, move, delete, z-order)
- Basic node types: solid color, text/markdown, static image, interactive hit region
- Sync groups (basic membership, no media clock yet)
- Atomic batch mutations (staged, committed per-frame, all-or-nothing)
- Zone system: named publishing surfaces with geometry policies, media type constraints, and rendering opinions. V1 ships at least: subtitle, notification, status-bar, and ambient-background zones. Agents can publish to zones with a single call — no tile/geometry management required.

### Compositor
- wgpu headless and windowed rendering
- Tile composition with z-order
- Alpha blending for overlays
- Background, tile borders, basic visual chrome
- 60fps on reference hardware

### Protocol
- gRPC control plane with protobuf
  - Scene mutation RPCs (create/update/delete tiles, switch tabs)
  - Lease management RPCs (request, renew, revoke)
  - Event subscription stream (scene changes, input events)
  - Telemetry stream
- MCP compatibility layer
  - Basic tool set: create_tab, create_tile, set_content, dismiss, list_scene
  - Zone tools: publish_to_zone, list_zones — the primary LLM-first publishing surface via MCP

### Security
- Agent authentication (pluggable, with at least pre-shared key and local socket implementations)
- Capability scopes (additive grants, revocation)
- Agent isolation (no cross-agent content access)
- Resource budgets (enforced, with throttle and revoke)

### Interaction
- Mouse/pointer input with hit testing
- Touch input on supported platforms
- Local-first feedback (press states, focus rings, hover highlights)
- Input events forwarded to owning agent
- At least one retained interactive primitive: hit_region node with local pressed/hovered state, focus semantics, and agent callback on activation. Without this, "local-first interaction" cannot be demonstrated and agents will fake interactivity by compositing images.

### Window modes
- Fullscreen mode: guaranteed on all platforms. Compositor owns the entire display, opaque background.
- Overlay mode (HUD): transparent always-on-top window with selective click-through. Supported on Windows, macOS, X11, wlroots Wayland. Falls back to fullscreen on unsupported compositors.
- Per-region input routing: tiles with active leases capture input, everything else passes through
- Runtime configuration, not compile-time — same binary supports both modes

### Platform
- Linux (X11 and Wayland), Windows (Win32), and macOS (Cocoa)
- wgpu backend: Vulkan on Linux, D3D12 (primary) + Vulkan (fallback) on Windows, Metal on macOS
- Headless CI: mesa llvmpipe on Linux, WARP on Windows, hardware-backed macOS CI runners (no proven software-only Metal path exists yet — revisit if one emerges)
- All three platforms are first-class — tested in CI, not "happens to compile"
- Overlay mode click-through: WM_NCHITTEST on Windows, hitTest on macOS, XShape on X11, wlr-layer-shell on Wayland (wlroots compositors only — GNOME/KDE Wayland falls back to fullscreen)

### Failure handling
- Agent disconnect detection with grace period
- Lease orphaning and cleanup
- Disconnection visual indicator
- Reconnection with lease reclaim

### Validation
- All five layers operational
- Test scene registry with initial corpus
- Hardware calibration and normalized benchmarks
- Developer visibility artifact pipeline
- Property-based testing for scene graph

### Telemetry
- Per-frame structured telemetry (timing, throughput, resources, correctness)
- Per-session aggregates
- JSON emission for CI consumption

## V1 explicitly defers

### Media
- No GStreamer integration. No video decode, no live camera feeds, no media pipelines. Tiles in v1 display static content (text, images, solid colors), not live media.
- No WebRTC. No audio/video streaming.
- No clocked media/cue message class. Sync groups exist but only for scene-level coordination, not media timing.

### Mobile
- No mobile build target. V1 is desktop/server only (Linux, Windows, macOS).
- No upstream precomposition.
- No mobile-specific degradation policies beyond what the general degradation ladder provides.
- Mobile capability negotiation is designed into the API but not exercised.

### Advanced presence
- No embodied presence level. V1 has guest and resident only.
- No dynamic orchestrator agent. Multi-agent coordination is flat — each agent negotiates independently with the runtime. Zone definitions in v1 are static (loaded from configuration at startup), not dynamically orchestrated by an agent.
- No inter-agent event bus beyond basic scene topology changes.

### Advanced rendering
- No browser surfaces embedded in tiles.
- No animated transitions between scenes or tabs.
- No particle effects, shaders, or custom render pipelines.
- No canvas node type (freeform drawing).

### Advanced protocol
- No WebTransport.
- No gRPC-Web for browser clients.
- No resumable state sync (reconnecting agents get a full snapshot, not a diff).

### Persistence
- No scene persistence across display node restarts (tab/layout config and auth registrations do persist).
- No durable scene snapshots or replay.

### Tooling
- No TypeScript inspector or admin panel.
- V1 does ship a minimal CLI/gRPC introspection surface for debugging: scene graph dump, active lease listing, resource utilization per-agent, zone registry state, and telemetry snapshot. This is not a GUI — it is a programmatic debugging surface usable by LLMs and developers via command line or gRPC reflection. Full remote inspection tooling is deferred.

## Why this boundary

The deferred items are all real requirements. They are deferred because:

**Media is the hardest integration** and it compounds every other problem (timing, synchronization, resource management, decode pipeline). Adding it to v1 would double the surface area without first proving that the core scene model, lease system, and compositor work correctly. Media is the first post-v1 priority.

**Mobile requires the core to be stable.** Negotiating capabilities and budgets is meaningless if the underlying system doesn't work reliably yet. Mobile is designed into the API from day one but exercised after the desktop path is proven.

**Advanced features build on the core.** Transitions, browser surfaces, orchestration, persistence — they all assume a working compositor with a working lease model. Adding them prematurely obscures whether the foundation is solid.

The principle: v1 proves the thesis with the minimum surface area. Everything deferred is deferred because it depends on v1 being right, not because it's unimportant.

## V1 success criteria

V1 is done when:

- A human can watch 3 LLM agents simultaneously hold tiles on a screen, updating content in real time, governed by leases, at 60fps.
- An LLM can autonomously: write a new tile type, test it across all five validation layers, open a PR with developer visibility artifacts, and have the PR be mergeable.
- The security model prevents a rogue agent from affecting other agents or the runtime.
- An LLM with only MCP access can publish a subtitle to a zone with one tool call and see it render correctly — zero scene context, zero tile management.
- The whole thing runs headlessly in CI on Linux (llvmpipe), Windows (WARP), and macOS (Metal) with no manual intervention.

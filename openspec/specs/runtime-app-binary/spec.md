# runtime-app-binary Specification
Status: implemented

## Purpose
Defines the canonical operator-facing runtime application binary, its configuration-driven startup behavior, and stable Windows artifact identity for automation.

Implementation: app/tze_hud_app/

## Requirements
### Requirement: Canonical Runtime Application Executable
The project SHALL provide a canonical, non-demo runtime application executable target distinct from example/demo binaries. The canonical executable SHALL be documented as the primary operator-facing runtime artifact.

#### Scenario: Runtime executable target is discoverable
- **WHEN** build metadata is queried for binary targets
- **THEN** exactly one canonical runtime app target is identifiable as non-demo
- **AND** demo/example targets remain available but are not designated as the primary runtime artifact

### Requirement: Configuration-Driven Runtime Startup
The canonical runtime executable SHALL support configuration-driven startup for window mode and network endpoint settings, including deterministic enable/disable behavior for each endpoint.

#### Scenario: Runtime starts with configured display and network settings
- **WHEN** the runtime executable is launched with a valid runtime configuration
- **THEN** it SHALL apply configured window mode and dimensions
- **AND** it SHALL enable only the configured network endpoints
- **AND** it SHALL keep disabled endpoints unbound

### Requirement: Windows Artifact Identity for Automation
The canonical runtime executable SHALL produce a deterministic Windows build artifact identity suitable for deployment automation.

#### Scenario: Automation resolves artifact path without heuristics
- **WHEN** the runtime executable is built for a Windows target triple
- **THEN** the artifact name and output location SHALL be stable and documented
- **AND** deployment automation SHALL be able to reference the artifact without relying on demo-target fallbacks

---

### Requirement: Release Artifact Provenance

Each published Windows release artifact (`tze_hud.exe`) SHALL be accompanied by a
SHA-256 checksum generated by the release pipeline, not by a manual operator step. The
published checksum MUST reference the deterministic artifact name `tze_hud.exe`.
Deployment automation MUST be able to verify the artifact against the published checksum
before activation, and MUST refuse to activate the artifact on a checksum mismatch. The
checksum generation MUST be performed in CI, and the verification procedure MUST be
documented.

Cryptographic signing of the artifact (for example cosign or gpg) is OPTIONAL and MAY be
deferred beyond v1; when signing is added, the published signature MUST cover the same
deterministic artifact and the verification procedure MUST be extended to check it.

Source: `README.md` §1.2, `.github/workflows/ci.yml`, `scripts/ci/release_artifact_provenance_smoke.sh`
Scope: v1-mandatory

#### Scenario: Checksum published with artifact

- **WHEN** the release pipeline produces `tze_hud.exe`
- **THEN** it SHALL also produce and publish a SHA-256 checksum that references the artifact name `tze_hud.exe`
- **AND** the checksum SHALL be generated by the pipeline, not by a manual operator step

#### Scenario: Verification before activation

- **WHEN** deployment automation receives `tze_hud.exe` and its published SHA-256 checksum
- **THEN** it SHALL compute the artifact's SHA-256 and compare it against the published checksum
- **AND** it SHALL refuse to activate the artifact if the computed checksum does not match

#### Scenario: Signing is optional and deferred

- **WHEN** v1 ships without cryptographic signing
- **THEN** the provenance requirement SHALL still be satisfied by the published SHA-256 checksum alone
- **AND** the absence of a signature SHALL NOT block release activation in v1

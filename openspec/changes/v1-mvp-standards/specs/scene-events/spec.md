# Scene Events Specification

## ADDED Requirements

### Requirement: Three-Category Event Taxonomy
The runtime SHALL classify all events into exactly three categories: input events (owned by RFC 0004), scene events (this specification), and system events (runtime-only, limited agent visibility). Each SceneEvent variant SHALL belong to exactly one subscription category. The runtime SHALL NOT deliver events in categories the agent has not subscribed to or lacks capability for.
Source: RFC 0010 §1.1, §1.2
Scope: v1-mandatory

#### Scenario: Event taxonomy completeness
- **WHEN** any event is generated by the runtime, an agent, or a system subsystem
- **THEN** it MUST belong to exactly one of the three categories (input, scene, system) and map to exactly one subscription category

#### Scenario: Unsubscribed category filtering
- **WHEN** an agent has not subscribed to `scene_topology`
- **THEN** the runtime MUST NOT deliver TileCreated, TileDeleted, TabCreated, AgentJoined, or any other scene_topology events to that agent

---

### Requirement: SceneEvent Envelope Structure
Every scene event SHALL carry a common envelope containing: event_id (UUID v7, 16 bytes), event_type (namespaced dotted string), interruption_class (effective class after ceiling enforcement), timestamp_wall_us, timestamp_mono_us, source_lease_id (empty for system events), source_namespace (empty for runtime events), and sequence (monotonic scene sequence number for gap detection). The payload SHALL be a oneof of zone, tile, tab, agent, system, or sync_group payloads.
Source: RFC 0010 §2.1
Scope: v1-mandatory

#### Scenario: Event envelope fields populated
- **WHEN** the runtime generates a TileCreated scene event
- **THEN** the SceneEvent envelope MUST contain a non-zero UUID v7 event_id, event_type "scene.tile.created", a valid InterruptionClass, both wall and monotonic timestamps, the source lease_id, the source agent namespace, and a monotonically increasing sequence number

#### Scenario: System event source fields
- **WHEN** a system event (e.g., degradation_changed) is generated
- **THEN** source_lease_id MUST be empty and source_namespace MUST be empty

---

### Requirement: Event Type Naming Convention
Event types SHALL use a dotted namespace hierarchy: scene events as `scene.<object>.<action>`, agent events as `agent.<namespace>.<category>.<action>`, and system events as `system.<action>`. All segments SHALL use only lowercase letters, digits, and underscores. The prefixes `system.` and `scene.` SHALL be reserved for runtime-generated events; agents MUST NOT emit events with these prefixes. Input event type naming is governed by RFC 0004 and is outside the scope of this requirement.
Source: RFC 0010 §2.2
Scope: v1-mandatory

#### Scenario: Agent event namespace prefixing
- **WHEN** an agent with namespace "doorbell_agent" emits event "doorbell.ring"
- **THEN** the delivered event_type MUST be "agent.doorbell_agent.doorbell.ring"

#### Scenario: Reserved prefix rejection
- **WHEN** an agent attempts to emit an event with name starting with "system." or "scene."
- **THEN** the runtime MUST reject the emission

---

### Requirement: Interruption Classification
Every event that produces visible output or demands viewer attention SHALL carry an effective InterruptionClass. The five wire-level classes are: CRITICAL (overrides everything, bypasses quiet hours and budget), HIGH (may override quiet hours depending on pass_through_class, subject to budget; maps to "Urgent" in privacy.md doctrine), NORMAL (standard, filtered by attention budget), LOW (batched/deferred, blocked during quiet hours, subtle indicators only, counted; maps to "Gentle" in privacy.md doctrine), and SILENT (never interrupts, always passes quiet hours). Lower numeric value SHALL represent higher urgency. The runtime SHALL always apply the more restrictive of the agent-declared class and the zone's ceiling class.
Source: RFC 0010 §3.1, §3.2
Scope: v1-mandatory

#### Scenario: Ceiling enforcement
- **WHEN** an agent declares InterruptionClass HIGH on a zone whose ceiling_interruption_class is NORMAL
- **THEN** the effective interruption class MUST be NORMAL (the more restrictive of the two)

#### Scenario: CRITICAL agent restriction
- **WHEN** an agent requests CRITICAL interruption class on an emitted event
- **THEN** the runtime MUST downgrade the effective class to HIGH, since only the runtime can emit CRITICAL events

#### Scenario: SILENT always passes
- **WHEN** quiet hours are active and an event has InterruptionClass SILENT
- **THEN** the event MUST be delivered immediately (silent events are invisible by definition)

---

### Requirement: Quiet Hours Enforcement
During configured quiet hours, the interruption gate SHALL evaluate each event's interruption class against the pass_through_class threshold. CRITICAL events SHALL always be delivered and never queued. HIGH events SHALL pass through quiet hours when pass_through_class is HIGH or lower (this is the default configuration); otherwise HIGH events are deferred to the queue. NORMAL events SHALL be queued and delivered when quiet hours end. LOW events SHALL be discarded (they are too stale by quiet hours exit to be useful). SILENT events SHALL be unaffected. Quiet hours SHALL affect delivery, not generation; events SHALL still be created, logged, and counted for telemetry.
Source: RFC 0010 §4.1, §4.2, §4.3
Scope: v1-mandatory

#### Scenario: CRITICAL bypasses quiet hours
- **WHEN** quiet hours are active and a CRITICAL event (e.g., safe mode) is generated
- **THEN** the event MUST be delivered immediately without queuing

#### Scenario: HIGH passes quiet hours (default configuration)
- **WHEN** quiet hours are active with default pass_through_class=HIGH, and a HIGH event (e.g., doorbell) is generated
- **THEN** the event MUST be delivered immediately (HIGH passes through quiet hours when pass_through_class <= HIGH)

#### Scenario: NORMAL deferred during quiet hours
- **WHEN** quiet hours are active and a NORMAL zone update event occurs
- **THEN** the event's visual change MUST be queued and delivered when quiet hours end

#### Scenario: LOW discarded during quiet hours
- **WHEN** quiet hours are active and a LOW-class event is generated
- **THEN** the event MUST be discarded (not queued); LOW events are too stale by quiet hours exit to be useful

---

### Requirement: Quiet Hours Queue Semantics
Queued events SHALL accumulate in a per-zone FIFO queue. On quiet hours exit, the runtime SHALL dequeue and deliver them in FIFO order. Queued zone publishes SHALL follow their zone's contention policy (e.g., LatestWins delivers only the last). The queue SHALL have a configurable maximum depth per zone (default: 100). Events exceeding the queue depth SHALL be dropped oldest-first.
Source: RFC 0010 §4.3
Scope: v1-mandatory

#### Scenario: LatestWins zone coalesces queued publishes
- **WHEN** quiet hours are active and a LatestWins zone receives 10 queued publishes
- **THEN** on quiet hours exit, only the last publish MUST be delivered

#### Scenario: Queue depth limit
- **WHEN** the quiet hours queue for a zone reaches 100 entries and a new event arrives
- **THEN** the oldest entry MUST be dropped to make room for the new event

---

### Requirement: Agent Event Emission Protocol
Agents SHALL emit scene events via the EmitSceneEvent message on the session stream. Emission SHALL require the `emit_scene_event:<event_name>` capability, where `<event_name>` is the agent-supplied `<category>.<action>` suffix (e.g., `emit_scene_event:doorbell.ring`), not the fully-prefixed delivered event_type. The runtime SHALL namespace-prefix agent events as `agent.<namespace>.<category>.<action>` before delivery. Agent event payload SHALL be limited to 4KB maximum.
Source: RFC 0010 §5.1, §5.2
Scope: v1-mandatory

#### Scenario: Capability-gated emission
- **WHEN** an agent without `emit_scene_event:doorbell.ring` capability attempts to emit "doorbell.ring"
- **THEN** the runtime MUST reject the emission with an error in EmitSceneEventResult

#### Scenario: Successful emission with namespace prefix
- **WHEN** an agent with namespace "alarm_agent" and `emit_scene_event:fire.detected` capability emits "fire.detected"
- **THEN** the event MUST be delivered to subscribers with event_type "agent.alarm_agent.fire.detected"

#### Scenario: Payload size limit
- **WHEN** an agent emits an event with a payload exceeding 4KB
- **THEN** the runtime MUST reject the emission

---

### Requirement: Agent Event Rate Limiting
Agent event emission SHALL be rate limited to a configurable maximum (default: 10 events/second per agent session). Rate limiting SHALL use a sliding window (1-second window, max count). Events exceeding the rate limit SHALL be rejected with RuntimeError code AGENT_EVENT_RATE_EXCEEDED.
Source: RFC 0010 §5.4
Scope: v1-mandatory

#### Scenario: Rate limit enforcement
- **WHEN** an agent emits 11 events within a 1-second sliding window (default limit: 10/s)
- **THEN** the 11th event MUST be rejected with AGENT_EVENT_RATE_EXCEEDED

---

### Requirement: Attention Budget Enforcement
The runtime SHALL maintain per-agent and per-zone attention budgets as rolling counts of non-silent interruptions. Per-agent default: 20 interruptions per minute. Per-zone default: 10 per minute (30 for Stack-policy zones). CRITICAL interruptions SHALL be exempt from budget. HIGH interruptions SHALL pass through quiet hours (by default) but remain subject to budget (even valid HIGH-class interruptions may be coalesced or deferred if the attention budget is exhausted). When either counter reaches 80% of its limit, the runtime SHALL emit AttentionBudgetWarningEvent. When budget is exhausted, mutations SHALL proceed but their visual presentation SHALL be coalesced (latest-wins within the coalesce buffer). SILENT mutations SHALL carry zero interruption cost.
Source: RFC 0010 §6.1, §6.2, §6.3
Scope: v1-mandatory

#### Scenario: Budget warning at 80%
- **WHEN** an agent's rolling interruption count reaches 16 (80% of default 20 per minute)
- **THEN** the runtime MUST emit AttentionBudgetWarningEvent to agents subscribed to attention_events

#### Scenario: Budget exhaustion coalescing
- **WHEN** an agent exceeds its per-agent interruption budget and submits another NORMAL mutation
- **THEN** the mutation MUST commit normally but its visual presentation MUST be coalesced

#### Scenario: CRITICAL exempt from budget
- **WHEN** a CRITICAL-class event is generated while the budget is exhausted
- **THEN** the event MUST be delivered immediately without coalescing

---

### Requirement: Subscription Model with Category Filtering
Agents SHALL subscribe to event categories via SubscriptionChange messages. The complete category set SHALL include: SCENE_TOPOLOGY, INPUT_EVENTS, FOCUS_EVENTS, DEGRADATION_NOTICES (cannot opt out), LEASE_CHANGES (cannot opt out), ZONE_EVENTS, TELEMETRY_FRAMES, ATTENTION_EVENTS, and AGENT_EVENTS. Within each category, agents MAY apply finer-grained filters using event type prefix matching with trailing wildcards (e.g., "scene.*"). Maximum subscriptions per agent SHALL be 32. The runtime SHALL use the following mapping between subscription category names and event type prefix patterns:

| Subscription Category (SHOUTY_SNAKE_CASE) | Event Type Prefix Pattern |
|-------------------------------------------|--------------------------|
| SCENE_TOPOLOGY | `scene.*` |
| LEASE_CHANGES | `system.lease_*` |
| DEGRADATION_NOTICES | `system.degradation_*` |
| ZONE_EVENTS | `scene.zone.*` |
| FOCUS_EVENTS | `scene.focus.*` |
| INPUT_EVENTS | `input.*` (governed by RFC 0004) |
| AGENT_EVENTS | `agent.*` |
| TELEMETRY_FRAMES | `system.telemetry_*` |
| ATTENTION_EVENTS | `system.attention_*` |
Source: RFC 0010 §7.1, §7.2, §7.3
Scope: v1-mandatory

#### Scenario: Mandatory subscription categories
- **WHEN** an agent attempts to unsubscribe from LEASE_CHANGES or DEGRADATION_NOTICES
- **THEN** the runtime MUST reject the unsubscription; these categories cannot be opted out of

#### Scenario: Event filter prefix matching
- **WHEN** an agent subscribes to scene_topology with filter "scene.zone.*"
- **THEN** the agent MUST receive ZoneOccupancyChanged events but MUST NOT receive TileCreated or ActiveTabChanged events

#### Scenario: Subscription limit
- **WHEN** an agent attempts to create a 33rd subscription (exceeding the limit of 32)
- **THEN** the excess subscriptions MUST be denied in SubscriptionChangeResult.denied_subscriptions

---

### Requirement: Event Bus Four-Stage Pipeline
Events SHALL flow through a four-stage pipeline: (1) Emission from runtime, agent, or system source; (2) Classification assigning effective InterruptionClass (< 5us); (3) Policy Filtering evaluating quiet hours and attention budget; (4) Delivery fan-out to subscribed agents (< 100us per subscriber).
Source: RFC 0010 §8.1
Scope: v1-mandatory

#### Scenario: Pipeline stage ordering
- **WHEN** an agent emits a scene event
- **THEN** the event MUST pass through emission, classification, policy filtering, and delivery stages in that order before reaching any subscriber

---

### Requirement: Event Classification Latency
Event classification (Stage 2 of the pipeline) SHALL complete in less than 5 microseconds per event. This budget ensures classification does not contribute meaningfully to the 16.6ms frame budget at 60fps.
Source: RFC 0010 §8.3, §12
Scope: v1-mandatory

#### Scenario: Classification latency budget
- **WHEN** any event passes through the classification stage
- **THEN** the classification MUST complete within 5 microseconds

---

### Requirement: Event Delivery Latency
Event delivery to a subscriber (Stage 4) SHALL complete in less than 100 microseconds from emission. Subscribers SHALL observe scene changes within the same frame they occur.
Source: RFC 0010 §8.5, §12
Scope: v1-mandatory

#### Scenario: Delivery latency within budget
- **WHEN** a TileCreated event is emitted during the compositor's mutation commit
- **THEN** the event MUST be enqueued on each matching subscriber's outbound stream within 100 microseconds of emission

---

### Requirement: Event Coalescing Under Backpressure
Under backpressure, the runtime SHALL coalesce events with the same coalesce key. Each event type SHALL have a defined coalesce key (e.g., tile_id for TileUpdated, singleton for ActiveTabChanged). Lease and degradation events SHALL never be dropped, even under extreme backpressure. The coalesce buffer per subscriber SHALL be bounded to 64 entries.
Source: RFC 0010 §8.5, §8.6
Scope: v1-mandatory

#### Scenario: TileUpdated coalescing
- **WHEN** a subscriber's outbound buffer is full and multiple TileUpdated events arrive for the same tile_id
- **THEN** only the latest TileUpdated for that tile_id MUST be retained; earlier updates MUST be coalesced away

#### Scenario: Lease events never dropped
- **WHEN** a subscriber is under extreme backpressure and a LeaseRevokedEvent arrives
- **THEN** the LeaseRevokedEvent MUST NOT be dropped or coalesced away

---

### Requirement: tab_switch_on_event Contract
The runtime SHALL support a `tab_switch_on_event` configuration field per tab that names a scene-level event triggering automatic tab activation. Agent events SHALL match against the bare event name (before namespace prefixing) for agent-independence. System events SHALL NOT trigger tab switches. The triggered tab switch SHALL be subject to attention filtering (quiet hours, attention budget). A successful switch SHALL generate an ActiveTabChangedEvent.
Source: RFC 0010 §9.1, §9.2, §9.3, §9.4
Scope: v1-mandatory

#### Scenario: Agent-independent tab switch
- **WHEN** a tab is configured with `tab_switch_on_event = "doorbell.ring"` and any agent with the capability emits "doorbell.ring"
- **THEN** the runtime MUST switch to that tab (subject to attention filtering) regardless of which agent emitted the event

#### Scenario: Quiet hours deferral of tab switch
- **WHEN** quiet hours are active and a tab_switch_on_event fires with NORMAL interruption class
- **THEN** the tab switch MUST be deferred until quiet hours end

#### Scenario: System events excluded
- **WHEN** a tab is configured with `tab_switch_on_event = "system.degradation_changed"`
- **THEN** the runtime MUST NOT trigger a tab switch from system events; the system.* prefix is excluded from tab_switch_on_event matching

---

### Requirement: Self-Event Suppression
The runtime MUST NOT deliver a SceneEvent to the agent whose MutationBatch caused the state change. Lease and degradation events SHALL be exempt from self-event suppression and SHALL always be delivered even if the agent's own action triggered them. Self-event suppression check SHALL complete in less than 1 microsecond per event.
Source: RFC 0010 §14.2, §12
Scope: v1-mandatory

#### Scenario: Self-event not delivered
- **WHEN** Agent A commits a MutationBatch that creates a tile
- **THEN** Agent A MUST NOT receive the TileCreated event; it has ground truth from BatchCommitted

#### Scenario: Lease revocation self-delivery
- **WHEN** Agent A's budget violation triggers revocation of Agent A's own lease
- **THEN** Agent A MUST still receive the LeaseRevokedEvent so it knows its lease is gone

---

### Requirement: Event Ordering and Sequence Numbers
Events within a session SHALL be delivered in generation order, indexed by the SceneEvent.sequence field. An agent that misses events due to disconnection SHALL receive a full SceneSnapshot on reconnect and SHALL NOT receive a replay of missed events. The sequence field SHALL allow agents to detect gaps indicating a reconnect+snapshot cycle is needed.
Source: RFC 0010 §14.1
Scope: v1-mandatory

#### Scenario: Gap detection
- **WHEN** an agent receives events with sequence numbers 10, 11, 14 (gap at 12-13)
- **THEN** the agent MUST detect the gap and request a SceneSnapshot for reconnect

---

### Requirement: Audit Event Visibility Restrictions
Shell-state audit events from RFC 0007 (safe mode, freeze, override actions) SHALL be system events with CRITICAL interruption class. They SHALL always be logged to the audit trail regardless of subscription. Most audit events SHALL NOT be delivered to agents. Agents SHALL receive downstream effects (LeaseSuspended, LeaseResumed, DegradationLevelChanged) but SHALL NOT receive the raw override commands (freeze, dismiss, mute) to preserve viewer privacy.
Source: RFC 0010 §10.1, §10.2
Scope: v1-mandatory

#### Scenario: Safe mode experienced as lease suspension
- **WHEN** the viewer enters safe mode
- **THEN** agents MUST receive LeaseSuspendedEvent for their leases but MUST NOT receive the system.safe_mode_entered event

#### Scenario: Freeze invisible to agents
- **WHEN** the scene is frozen by the viewer
- **THEN** agents MUST NOT be informed of the freeze; their mutations SHALL be silently queued

---

### Requirement: Aggregate Event Rate Cap
The maximum aggregate event rate across all sources SHALL be 1000 events/second. Above this rate, the system is considered misbehaving. Individual agent sources SHALL be limited more tightly (10 events/second per agent by default).
Source: RFC 0010 §12
Scope: v1-mandatory

#### Scenario: Aggregate rate exceeded
- **WHEN** total event generation across all sources exceeds 1000 events/second sustained
- **THEN** the runtime MUST apply backpressure or shedding to bring the rate within budget

---

### Requirement: ZoneOccupancyChanged Dual Routing with Deduplication
The ZoneOccupancyChanged event SHALL appear in both scene_topology (for topology subscribers) and zone_events (for agents with zone publish permission). An agent subscribing to both categories SHALL receive the event exactly once; the runtime MUST deduplicate before delivery.
Source: RFC 0010 §1.2
Scope: v1-mandatory

#### Scenario: Deduplication across categories
- **WHEN** an agent subscribes to both scene_topology and zone_events, and a ZoneOccupancyChanged event occurs
- **THEN** the agent MUST receive the event exactly once, not twice

---

### Requirement: Earned Urgency Tracking
The runtime SHALL track per-agent HIGH-class event rates. When the rate exceeds the configurable threshold (default: 4 HIGH per agent per minute), the runtime SHALL log a warning. This implements the "earned urgency" principle from attention.md: "an agent that escalates everything is an agent that means nothing." Per-HIGH-class budget enforcement is deferred to post-v1.
Source: RFC 0010 §3.3
Scope: v1-mandatory

#### Scenario: HIGH-class rate warning
- **WHEN** an agent emits 5 HIGH-class events within one minute (exceeding default threshold of 4)
- **THEN** the runtime MUST log a warning about the agent's disproportionate HIGH-class usage

---

### Requirement: Inter-Agent Event Bus
A full inter-agent event bus enabling direct agent-to-agent event routing beyond the scene-level coordination signals is deferred.
Source: RFC 0010 §1.1, v1.md §"V1 explicitly defers"
Scope: post-v1

#### Scenario: Deferred marker
- **WHEN** an agent attempts direct agent-to-agent event messaging
- **THEN** this capability is not available in v1; agents coordinate only through scene-level events

---

### Requirement: Advanced Event Routing
Advanced event routing capabilities (priority queues, content-based routing, event transformations) are deferred beyond v1.
Source: v1.md §"V1 explicitly defers"
Scope: post-v1

#### Scenario: Deferred marker
- **WHEN** advanced routing semantics are needed
- **THEN** v1 provides only the four-stage pipeline with prefix-match filtering; advanced routing is post-v1

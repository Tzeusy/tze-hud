# V2 Program

> **DEFERRED INDEFINITELY (2026-05-09).** The v2 program — media plane, embodied presence, mobile/glasses device profiles, bidirectional AV, cloud-relay, recording, agent-to-agent media — is parked. The project has refocused on a performant single-device Rust HUD runtime for Windows. The 44 `hud-ora8.*` beads that operationalized this doctrine were closed; the openspec change moved to `openspec/changes/_deferred/v2-embodied-media-presence/`. Active scope and bead spawning are governed by `openspec/changes/windows-first-performant-runtime/` and epic `hud-9wljr`. **Do not act on this doctrine.** It remains here for historical reference; multi-device returns only after the single-Windows runtime is done.
>
> Original v2 doctrine follows.

---

V1 proves the thesis: a governed presence engine, a working lease model, a
compositor at 60fps, multiple agents coexisting on a real screen. That is
enough to prove the thesis. It is not enough to realize it.

The full thesis requires a presence engine that hears, speaks, and sees — that
can be embodied, not just rendered. V2 is the program that closes that gap.

This document states what the V2 Program is, what its four phases are, and
what V2 explicitly is not. It does not specify wire formats, field numbers, or
state machines — those belong to the RFC layer. This is doctrine.

## What the V2 Program is

V2 is a four-phase program that extends tze_hud from a v1 text-and-scene
runtime into a full multi-plane presence engine. It activates capabilities that
v1 deliberately deferred: live media, embodied presence, real device profiles,
and the validation infrastructure to operate all of it at production quality.

V2 is not a parallel system. It is not a rewrite. It extends the same runtime,
the same scene model, the same lease governance, the same compositing thread
that v1 proved. Every governance invariant from v1 — no agent in the frame
loop, arrival time is not presentation time, the screen is sovereign — holds
without relaxation. V2 adds planes; it does not revoke foundations.

## The four phases

**Phase 1 — Media plane activation.** The media plane (WebRTC + GStreamer)
activates as a real runtime capability, starting from bounded ingress and
extending through the operator-governance stack. Phase 1 delivers: media
admission gated by capability dialogs (per RFC 0008 Amendment 1 / C13),
decoded frames delivered over bounded ring buffers to GPU-resident compositor
surfaces, media workers running as in-process tokio tasks under E24's budget
watchdog, and the E25 degradation ladder governing all media shedding. Phase 1
explicitly absorbs the existing bounded-ingress contract tranche rather than
treating it as a separate v1.1 track. The pre-existing
`media-webrtc-bounded-ingress` and `media-webrtc-privacy-operator-policy`
specs are superseded by the v2 media-plane spec at phase 1 archive time, with
a line-by-line preservation audit in the closeout report. The companion
doctrine for this phase is `media-doctrine.md`.

**Phase 2 — Embodied presence.** A third presence level activates above
resident. Embodied agents have durable session identity, mandatory
operator-visible governance, and strict revocation semantics (soft within 500
ms, hard within 100 ms with audit). The single-embodied-agent rule (A4) holds:
at most one embodied agent occupies the slot at any time. A second embodied
request is rejected, not queued. Multi-agent embodied presence is post-v2 and
must not be anticipated. Phase 2 also introduces the two new capability
namespaces — `identity-and-roles` (role definitions, user directory schema,
role-to-capability binding per RFC 0009 Amendment 1 / C12) and
`identity-portability` (device-reboot-persistent identity with
user-initiated cryptographic export/import for device migration, B9). Neither
federation enforcement nor multi-agent embodied cohesion ships in v2; the data
structures model federation-aware fields for post-v2 compatibility, but no
enforcement logic is wired.

**Phase 3 — Device-profile execution.** Mobile and glasses device profiles
become real deployment targets rather than schema-only placeholders. Mobile is
first-class in v2. Glasses are contract-ready: the upstream-composition wire
(pre-composited WebRTC frames plus input events upstream, operator policy
enforced at the compositor host) is generalized so external glasses SKUs can
plug in post-v2 without protocol breakage. Glasses are not exercised as a v2
validation target. Phase 3 overlaps the last month of phase 2 by design; the
sequencing is 1 → 2 → (3 overlaps late 2) → 4.

**Phase 4 — End-to-end validation and operations.** Bidirectional AV and voice
synthesis activate (sub-epic 4f), with cloud-relay trust boundary (4b),
recording with consent and retention governance (4a), and agent-to-agent media
routing via the runtime (4e). The internal phase-4 DAG is: {4f, 4b} first in
parallel, then {4a, 4e} in parallel, each blocked on the prior pair's closeout.
No phase-4 sub-epic opens until the phase-3 closeout report merges. Each
sub-epic requires its own RFC (0017 for recording and audit, 0018 for
cloud-relay trust boundary) and a closeout report following the v1 convention.

## What V2 IS

**An extension over v1, not a restart.** Every v1 invariant survives: the
single-process compositor, the agent-runtime wire as the trust boundary, the
lease model, the degradation ladder order, the human override contract. V2 adds
protocol planes and presence levels on top of that foundation.

**A media governance story, not just a media feature.** The media plane
activates under the same arbitration stack as everything else: capability gate
→ privacy/viewer gate → interruption policy → attention budget → degradation
budget. A video frame is not exempt from the governance that governs a
subtitle.

**A four-RFC-amendment event.** V2 lands through amendments to RFC 0002
(runtime-kernel, media worker lifecycle), RFC 0005 (session-protocol, embodied
flag and media signaling), RFC 0008 (lease-governance, capability dialogs and
7-day per-agent-per-capability memory), and RFC 0009 (policy arbitration,
role-based operators). New RFCs 0014 (media plane wire protocol) and 0015
(embodied presence contract) carry the mechanism layers for phases 1 and 2
respectively. RFCs 0016–0019 follow in phases 3 and 4. Doctrine precedes
RFC in each topic area, exactly as it did for v1.

## What V2 is NOT

**Not multi-agent embodied.** One embodied slot. Always. Multi-agent embodied
presence requires per-embodiment media plane isolation, independent focus
graphs, and cross-embodiment arbitration rules that the v2 protocol surface
does not validate. Any implementation that anticipates this capability is
acting outside doctrine.

**Not federation enforcement.** V2 models federation-aware fields in data
structures and reserves the `federated_presence_visibility` capability in the
schema. No federation enforcement logic ships. Cross-household/cross-operator
federation is v3 scope. Adding federation wire-logic inside a v2 phase is a
scope violation.

**Not background audio.** Media plays only when its owning surface is visible
or foregrounded. Background audio — media playing while the publishing
surface is hidden — is not admitted in v2. This is not a resource decision; it
is an attention governance decision. The attention model does not support
ambient audio for presences the user has not foregrounded.

**Not external transcoding services.** The transcoding abstraction points exist
in the media-plane spec. No vendor integration ships in v2. The use case must
emerge before the vendor choice locks.

## Sequencing and gates

V2 code does not land until v1 ships as a tagged release (`v1.0.0`) with
binary artifacts and a closeout reconciliation report. Doctrine and RFC work
for v2 proceeds in parallel with v1 finishing. Once v1 is tagged, v2 code
lands behind a `v2_preview` feature flag.

Each phase's first implementation bead is blocked until the prior phase's
closeout report merges. The re-baseline after phase 1 closes is a hard rule,
not a suggestion. Phase 2 kickoff is blocked on that re-baseline verdict.

The RFC sequence gates implementation. Six new RFCs (0014–0019) and four
amendments (0002, 0005, 0008, 0009) must merge before the implementation beads
they govern are created. RFC 0014 requires at least two external reviewers
given its fan-out across all later phases.
